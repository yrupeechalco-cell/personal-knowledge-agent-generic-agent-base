# 词云标签与知识域 v0.1

## 本版本目标

这一版把知识存储的基本单位从“文件夹中的文档”推进为“资料、标签、知识域”三层结构：

1. 资料仍然是原始 Markdown 文档，不改变用户已有的文件存储方式。
2. 每篇资料拥有一组可编辑的词云标签，用来表达资料真正涉及的知识点。
3. 全库标签被汇总成三维知识图谱，通过不同的“域”重新组织和观察。

这里的词云不是装饰性统计图。词云和标签是同一份数据，用户修改词云就等于修改资料的标签元数据，图谱也随之更新。

## 资料词云

- 标签来源包括用户已有标签、本地文本候选和模型提取结果。
- 颗粒度分为五档：概览、主题、标准、细分、术语。
- 颗粒度控制目标标签数量和提取细度，不修改正文。
- 标签支持新增、删除和重命名。
- 写入时沿用现有草稿与持久化流程；公开 GitHub 示例库保持只读。
- 模型不可用时使用本地提取器，不把“需要联网”伪装成必要条件。

Agent 提取遵循以下约束：

- 优先保留用户已经确认的标签。
- 只提取资料正文能够支持的概念，不凭空补充知识。
- 使用可复用、可分类的短标签，避免整句摘要充当标签。
- 模型输出经过结构化解析和数量限制后才进入资料元数据。

## 标签关系

当前版本只使用可以追溯的本地证据建立关系：

- 共现关系：两个标签出现在同一篇资料中。
- 显式链接关系：带有这些标签的资料之间存在 Markdown 双链。
- 来源关系：标签主要出现在哪个文件夹或资料集合中。

图谱不会把所有语义猜测直接当成事实。后续由 Agent 推断的关系应单独保存证据、置信度和人工确认状态。

## 四种知识域

“域”不是改变摄像机角度，而是改变同一批标签的组织规则和权重。

### 分类域

按照概念、方法、对象和证据四类组织标签。适合观察知识库由哪些基本类型组成，以及某一类是否过密或缺失。

### 脉络域

根据标签共现和文档显式链接形成物理布局。关联强的标签彼此靠近，同时保留整体空间，避免全部塌缩成一团。

### 应用域

根据标签覆盖资料数量、连接中心性和方法/行动语义调整位置与外圈明度。适合发现哪些标签更接近可执行知识，哪些只是背景资料。

### 来源域

按照标签的主要文件夹或资料集合分组。适合检查不同来源之间是否重复、孤立或产生跨来源联系。

## 图谱交互

- 三维图谱可旋转、平移和缩放。
- 同一个标签的基础球体大小在所有域中保持一致，表示它在全库中的总体分量。
- 总体分量只产生温和的尺寸差异；三维前后景会进行透视补偿，避免距离摄像机更近的球被重复放大。
- 域内重要性通过位置和外圈明度表达，不覆盖标签的总体分量。
- 同一条关系线的基础明度在所有域中保持一致；线长只来自空间布局，不作为关系强弱指标。
- 标签数量较多或中栏较窄时，只显示权重较高的部分文字；球体仍然保留。
- 鼠标悬停时，无关标签和连线会淡出，相关连线转为紫色。
- 点击标签球体后进入二维知识根系，按问题、证据、决策、成果和背景组织其来源资料。
- 点击根系叶子可直接进入对应笔记，返回箭头会恢复进入前的三维域。
- 全屏与窄窗口分别按中栏实际宽度调整标题、域切换和标签密度。

## 当前边界

- v0.1 的知识关系来自标签共现、显式双链和文件来源，不代表完整的知识因果关系。
- 词云粒度控制的是提取范围，不等同于自动判断知识真伪或重要性。
- 模型提取结果仍应允许用户编辑；Agent 是分类助手，不是不可纠正的分类权威。
- 公开 Web 示例库只用于观察，不能修改标签；本地可写库才会进入真实草稿和保存流程。

## 下一轮观察重点

1. 五档颗粒度是否足以覆盖短题目、长文章和多文档资料集。
2. 用户是否能理解四个域的差异，还是需要减少或重新命名。
3. 标签分量是否应加入用户手动权重，而不只依赖资料数量与连接。
4. Agent 推断的二层关系应如何展示证据和置信度，避免把猜测画成事实。
5. 是否需要资料集级词云，让一批文件共享上层标签而不重复编辑。
